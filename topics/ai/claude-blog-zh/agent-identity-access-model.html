Claude Tag 中的智能体身份：面向自主化、团队级 AI 的全新访问模型

原文标题：Agent identity in Claude Tag: a new access model for autonomous, team-wide AI ｜ 发布日期：2026 年 6 月 24 日 ｜ 分类：Claude Code ｜ 产品：Claude Tag ｜ 阅读时长：约 7 分钟 ｜ 查看原文

Claude Tag 的智能体身份访问模型如何运作，以及如何在你团队的工作空间中妥善配置它的最佳实践。

要让一个 AI 智能体在人机协作团队中发挥出最佳表现，它需要获得和人类一样的工具、文档和上下文访问权限。

在"单人"AI 体验中（一个人和一个助手对话），这很直接：你连接自己的账户，智能体代表你行事。但在像 Claude Tag 这样的"多人"AI 体验中，Claude 会和许多人同时坐在同一个共享频道里，它所依赖的工具和上下文，属于这个工作空间，而不是任何一个具体的人。

要让多人体验能够运转，Claude 需要为这些工具拥有自己的账户，由管理员设置并绑定到工作空间上。我们把这种访问模型称为智能体身份（agent identity）。

在这篇文章中，我们解释智能体身份是如何运作的，它如何把权限从"按用户"转变为"按频道"，以及如何在你自己的工作空间里对它进行妥善的范围界定。

为什么"代表用户行事"这套模式行不通了

当你把 AI 用作个人助理时，你可以连接 Google Drive、GitHub 和日历这类平台，让模型使用你自己的访问权限去读写这些内容。

这套模式对 Claude Tag 不适用，原因有两个：

  • 智能体自主性不断增强。 一个 AI 智能体能够可靠独立完成的任务时长，大约每四个月就翻一倍。智能体现在会为自己安排稍后执行的任务，并在提出请求的人早已下线之后，依然对事件做出响应。虽然用户设置了触发智能体在特定情况下采取行动的例行任务，但智能体在很大程度上是自主运作的。
  • 多人团队。 Claude Tag 把 Claude 放入团队已经在工作的共享空间中——比如一个由三名工程师和一名产品经理一起调试问题的频道。但当不止一个人在引导方向时，该套用谁的权限？没有哪一个人选是永远正确的。这让管理员能够独立于参与其中的具体人类，去定义一个智能体在 Slack 中可以做什么，并对它在 Slack 中所做的事情进行单独追踪。

Claude 以自身的身份行事

在一个启用了 Claude Tag 的频道里，Claude 并不是代表某一个用户行事。它在自己接触的每一个系统中都拥有自己的账户：它以 Claude 应用的身份在 Slack 中发帖，以 Claude GitHub App 的身份开启 pull request，以管理员配置的服务账户身份查询你的数据仓库。

而且，由于其中不涉及任何个人用户凭证，一个共享频道永远不可能变成通往某人私有文档的后门。

权限继承

在智能体身份模型中，管理员在工作空间层面定义一个身份——即 Claude 在任何地方都拥有的基线连接和技能集合——每个频道默认继承这个身份。然后，在合适的地方，他们可以在频道层面对其进行覆盖，比如让工程频道获得 GitHub 和数据仓库的访问权限，或者把某个 CRM 连接限定在一个私密频道内。

除了凭证之外，管理员还可以定义：

  • 仓库访问权限： Claude 可以读写哪些仓库。
  • 连接器： Claude 用来完成工作的工具和 API 密钥。在一个组织内部，不同的 API 密钥可以以不同的权限级别连接同一个服务（比如，Claude 在一个通用频道中可能只获得数据仓库的只读权限，而在数据团队的私密频道中则获得写入权限）。
  • 技能与插件： Claude 动态加载的指令、脚本和资源文件夹，用来提升它在专业任务上的表现。
  • 常设指令： 为每个频道定制的自定义指令和上下文。

由于这套模型是围绕独立的 Claude 身份构建的，撤销一个身份，就会终止 Claude 在使用过该身份的所有地方的访问权限。相比在数十个用户账户上逐一审计智能体的每一项操作，这样管理起来省事得多。

智能体身份模型是如何运作的

智能体身份把"这个用户能做什么"这个问题，替换成了"这个智能体在这个隔间里能做什么"。这和按用户设置的访问控制列表（ACL）是一种偏离：这意味着，一个没有直接权限访问某个仓库的频道成员，如果频道的配置文件授予了 Claude 相应权限，依然可以让 Claude 去读取那个仓库。

这种做法并不常见，但我们认为，这是迈向一套适用于自主化、多人智能体的访问模型所必需的一步。下面，我们会勾勒出该如何思考这些边界的设定方式。

身份边界是如何运作的

Claude Tag 会为每个私密频道创建一个独立的身份；一个工作空间内的公开频道则共享一个工作空间级别的身份。Claude 在法务频道中的身份，无法触及那里没有被授权的代码；它在工程频道中的身份，也无法读取那里没有被授权的法务文档。记忆和访问权限都遵循这些边界：Claude 在一个私密频道中学到的东西，永远不会出现在更广泛的工作空间里。

身份归属于频道本身，所以频道里的任何人默认都可以 @ Claude，管理员可以按照"最低权限成员"的标准来设定每个频道的配置文件范围。在 Enterprise 套餐中，基于角色的访问控制可以让管理员更进一步，决定哪些成员可以调用 Claude，因此一个频道既控制着智能体能触及什么，也控制着谁可以发起请求。

默认对工具和上下文拥有广泛的访问权限

在 Anthropic 内部运行 Claude Tag 的过程中，我们发现它的价值会随着工具和上下文访问权限的增加而复合式增长。每一个被连接的系统，都会让其他每一个系统变得更有用，因为 Claude 能够跨系统整合上下文——从 Slack 中提取一条消息线索，从 Drive 中取出一份文档，从工单系统中拿到一张工单，再从数据仓库中提取一条查询结果，汇总成一个任何单一工具都无法独立给出的答案。

从 Claude 身上获益最多的团队，是那些从一开始就给予它较为慷慨的访问权限、再根据组织的管理员偏好逐步收紧的团队。智能体身份给了管理员足够广泛的范围，让 Claude 能够完成有价值的跨系统工作，同时边界又足够牢固，确保访问权限永远不会流向未被授权的地方。我们的建议是：先在几个频道中设置一个基线配置文件，查看审计记录，然后根据工作需要逐步扩展访问权限，一次一项、谨慎授予。

对于需要更精细控制的组织，管理员可以在特定频道中禁用 Claude Tag。管理员也可以应用基于角色的访问控制（RBAC），把 Claude Tag 的使用权限限制在特定用户范围内。

私信

在 Claude Tag 中，私信的运作方式和共享频道不同。私信运行在用户各自的 claude.ai 账户上——使用他们各自的连接器、凭证，结果也以他们的名义呈现。这使得私信成为处理那些不应该出现在频道中的任务和工具的合适场所，比如邮件草稿，或者只有你自己拥有授权许可的软件。

安全与审计

当管理员向某个频道的配置文件添加一项连接时，对应的凭证会被独立存储，并映射到该频道的身份上，然后在请求发生时于网络边界处注入。任何管理员未允许访问的主机，其出站流量都会被直接拦截。在审计方面，每一次使用智能体凭证发起的例行任务、记忆写入和网络调用都会被记录下来，而且由于 Claude 是以自己的服务账户身份行动的，这些操作同样会出现在每个已连接系统自己的日志中。

接下来是什么

智能体身份是 Claude Tag 访问模型的基石。未来，我们计划进一步强化 Claude Tag 的安全能力，包括即时凭证授予——让用户能够当场批准某一次敏感操作，而不需要永久扩大智能体的权限范围——以及为拥有更复杂权限结构的组织提供一层身份感知的叠加机制。这将在智能体权限范围之上，再增加一层用户级别的检查，确保只有当频道的配置文件和发起请求用户自身的权限都同时允许时，Claude 才会采取行动。

像 Claude Tag 这样的产品所推动的"从单人到多人"AI 转变，让长时间运行、基于团队的工作成为可能。智能体身份确保了 Claude 对工具的访问既足够广泛、能够真正发挥作用，又在企业规模下受到足够的范围限制，保持安全。
