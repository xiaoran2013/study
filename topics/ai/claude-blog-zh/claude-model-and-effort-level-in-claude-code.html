在 Claude Code 中选择模型与努力级别（Effort Level）

原文标题：Choosing a Claude model and effort level in Claude Code ｜ 发布日期：2026 年 7 月 7 日 ｜ 分类：Claude Code ｜ 产品：Claude Code ｜ 阅读时长：约 5 分钟 ｜ 查看原文

要点速览

  • 选择 Claude 模型，本质上是选择一组固定的权重，也就是模型的整体能力范围。虽然模型可以被提供上下文、被引导，但它整体的知识库和能力是设定好的。
  • "努力（Effort）"的含义不止是"思考时间"。它控制的是 Claude 针对你的请求整体投入多少工作量，包括读取多少文件、使用多少工具，以及在回来找你确认之前要走多少步。
  • 常规任务选用较小的模型，复杂或含糊不清的任务选用较大的模型。先从每个模型的默认努力级别开始，再根据你所做工作的类型作为一种总体偏好去调整，而不是逐个任务地去调。
  • 如果 Claude 已经拥有全部相关上下文、也明显尝试过了，但结果依然错了，这就是该换一个能力更强的模型的信号。如果 Claude 出错的原因是跳过了某个文件、没有运行测试，或者在重构进行到一半时半途而废，那就应该选择更高的努力级别。

Claude Code 中的努力级别与模型选择

Claude Code 给了你两个看起来都能"让答案变得更好"的设置：模型设置和努力级别。你可能会以为，像 Claude Fable 5 这样更大的模型，会比 Claude Sonnet 给出更聪明的输出，而更高的努力级别意味着 Claude 在回答之前会思考更久。

第一个假设是对的。按照业界标准的基准测试，我们最大的模型能力确实更强。

但"努力"的含义不只是"思考时间"。努力级别控制的是 Claude 针对你的请求整体投入多少工作量。这确实包括模型思考多久，但也包括：

  • 它会读取多少个文件；
  • 它会做多少验证；以及
  • 在回来找你确认之前，它会把一个多步骤任务推进到多远。

在更高的努力级别下，Claude 在回来找你之前，会执行更多这类动作（比如读取文件、运行测试、反复核实）。而在更低的努力级别下，它更倾向于向你要更多上下文，而不是花费 token 自己去琢磨。

模型选择的原理

当你按下回车键，Claude Code 会把你的消息，连同系统提示词、工具定义、你的 CLAUDE.md、对话历史，以及上下文中的任何文件，一起组装起来。所有这些内容会作为一个请求发送给 API。

不过，模型看到的从来都不是纯文本。服务器端首先发生的事情是"分词（tokenization）"：文本被切分成一个个片段，每个片段会被映射到模型训练时所使用的固定词表中的一个整数。比如 const 可能映射为 1978，await 可能映射为 4293。从这一步开始，你的提示词就变成了一个整数数组。

模型的工作，就是拿到这个整数数组，然后预测下一个 token 会是什么。它的做法是为词表中的每一个 token 计算一个"概率"，然后从概率最高的那些里挑选。比如在 const x = await 之后，一个训练良好的模型会把很高的概率分配给 fetch（非常可能），而给 banana 分配几乎为零的概率（完全不可能）。

把你输入的 token 转变成这些概率的，正是"权重"（也叫"参数"）。这些是数十亿个数字，被组织成一个个巨大的矩阵。为了预测下一个 token，模型会让你的输入经过这些矩阵——一长串矩阵乘法运算——然后在最后读出各个概率。模型"知道"的一切，都存在于这些权重之中。

每个模型的权重都是在训练阶段确定的，等到你发送请求的那一刻，它们已经是只读的了。你的提示词、你的 CLAUDE.md，或者你提供的上下文，都不会改变这些权重。（如果你遇到过"推理（inference）"这个词，它指的正是这个：在训练完成之后使用模型，此时权重是固定不变的。）

Claude 对 TypeScript、各种流行框架、地道的 Go 语言写法，以及其他任何通用编程知识的了解，都是在训练阶段被编码进这些权重里的。

你的提示词和上下文依然可以"引导（steer）"预测结果（把你真实的代码摆在 Claude 面前就是一种引导，而且效果很好），但它们不会给权重本身增添任何东西。

如果某个库在模型训练时还不存在，它就不会出现在权重里。你可以把文档放进上下文，Claude 会使用它，但那是"引导"，不是"教学"。Claude 的回答只会在这次请求中受到影响；底层模型本身并不会保留这些信息。

所以当 Claude 信心满满地调用一个根本不存在的 API（也就是"幻觉"）时，这其实是权重根据训练时学到的模式，生成了一段"看起来说得通"的 token 序列，而不是一次查找失败。

那么，切换模型到底改变了什么？它换掉的是"由哪一组冻结的权重来处理你的请求"。

模型不会一次性生成整个答案。它每次预测一个 token，把它追加到序列末尾，然后重新跑一遍整个计算流程，去预测下一个 token。一个 200 个 token 的回答，意味着要经过 200 次独立的权重计算。这个循环，正是你大部分等待时间和输出成本的来源。

所以，"模型"这项设置决定的是由哪组权重来处理你的请求，同时也决定了每个输出 token 的成本。

但它不决定的是会生成多少个 token。对同一个提示词而言，这个数字可以有很大差异，这取决于 Claude 决定投入多少工作量。

而这正是"努力级别"所控制的：Claude 在每一轮对话中，决定投入多少工作量。

Claude Code 中努力级别的运作原理

当 Claude Code 在处理一项任务时，它生成的 token 大致可以分为几类：

  • 思考：你在各个操作之前、之间看到的、以流式方式呈现的推理过程。
  • 工具调用：指明某个工具（比如 Read 或 Edit）及其参数的结构化代码块，Claude Code 随后会解析并执行它。
  • 给你的文本：计划、进度更新，以及最后的总结。

这些其实都是同一个循环里产生的普通输出 token，按同样的费率计费。举例来说，思考型 token 的生成方式和其他输出 token 完全一样，并且会在这一轮对话剩余的时间里始终留在上下文中。

当 Claude 接下来去写代码时，它先前的推理过程会成为输入的一部分，就像它读过的某个文件一样。

那么"努力"到底改变了什么？努力级别会作为请求的一部分被发送给模型，就在你的提示词旁边一起送过去。模型被训练成能够理解在每个努力级别下应该如何表现，这种学到的行为方式，已经被固化进了冻结的权重之中。

当你的请求到达时，努力级别就是模型要响应的又一项输入，就和它对你提示词文本的响应方式一样。它决定了 Claude 在认为任务完成之前，需要做到多彻底、多有把握。

这个考量会在每一轮对话中都被纳入，其结果是需要更多的 token 才能给出更有把握的答案。

在更高的努力级别下，Claude 通常会先制定一个计划，努力级别的高低会影响这个计划的深度和广度。不过，这个计划并非一成不变。随着 Claude 陆续收到自己各项操作的结果，它会更新已经取得的进展，以及对累积结果的确信程度。

所以，如果一个包含三个假设的调试计划，在第一步就找到了 bug，那么"继续调查假设 2 和假设 3"可能就不再是必要的动作了。Claude 通常会明确说出这一点："第一项检查已经找到了问题，所以剩下的检查不需要了"，然后直接跳过。在 Claude Code 中，你会看到任务列表在运行过程中被临时修订，正是这种情况的体现。

在更高的努力级别下，Claude 会更倾向于对其他假设做双重检查，或者对正确性进行核实，但它通常不会为简单任务人为地虚增用量。事实上，我们的团队在模型训练时非常关注"过度思考"的问题，因为这会削弱效果。

如何选择努力级别

我们的建议是：大多数任务应该使用模型的默认努力级别。 默认级别是指 Claude 会根据大多数人愿意为这类任务投入多少而相应调整 token 用量的那个级别。

把"努力"想象成一个手动开关，用来调节 Claude 工作的强度和时长。当你基于自己的领域或工作类型，对"彻底程度"或"速度"有明确偏好时，可以有意识地去调整它。更应该把这当作一种总体偏好，而不是逐个任务去做的决定。

在 Claude Opus 4.8 发布之后，有一些实用的经验或许能帮你做判断：我们在测试中发现，在同样的 token 数量下，使用 Opus 4.8 的默认努力设置，比使用 Opus 4.7 的默认努力设置处理同一任务，能产出更好的结果。

你可以只为当前会话设置努力级别，也可以把它固化为默认值。对于单次会话，运行 /effort 可以打开一个滑块，运行 /effort high 可以直接设定某个级别，或者在启动时加上 --effort 参数。要让默认值在多次会话之间持续生效，可以在设置文件里配置 effortLevel，或者设置 CLAUDE_CODE_EFFORT_LEVEL 环境变量。如果同时设置了多个，环境变量的优先级最高，其次是设置文件中的值，最后才是模型自身的默认值。

有一点很容易让人踩坑：只有 low、medium、high 和 xhigh 可以被持久化保存。max 和 ultracode 只在当前会话中生效——每次想用的时候，都需要通过 /effort 或 --effort 单独设置。所以如果你把 max 写进了设置文件，却发现它似乎被忽略了或被悄悄降级了，原因就在这里。要确认当前实际生效的是哪个级别，可以查看加载动画（spinner）旁边显示的努力级别标签，或者打开 /model。

当 Claude 出错时，应该调整什么

当 Claude 出错的时候，你的第一反应不应该是去调某个旋钮，而应该是检查一下你提供的上下文。你的提示词是不是太模糊了？Claude 是否连接到了正确的工具？是否配备了合适的技能？

如果你在一个本不该需要更高努力级别的任务上不断调高努力，问题的根源往往在上游——出在你的上下文、你的 CLAUDE.md，或者任务的范围界定方式上。

但假设你已经提供了清晰的上下文，Claude 依然出了错，这时候该问自己的问题是：它是没有足够努力去尝试，还是本身就不够懂？

模型：问题本身太难

当问题确实很难时，选用更大的模型。比如那些隐晦的 bug、不熟悉的领域，或者架构决策类的问题。当较小的模型无论你给它多少上下文都会自信地给出错误答案时，更大的模型会很有帮助。

更大的模型也更擅长处理模糊性，而具体、明确的执行指令，则更适合在较小的模型上取得成功。

当工作是常规性的，选用较小的模型。比如你能精确描述的修改、机械性的改动，或者关于已经在上下文中的代码的问题。没有理由为一项任务不需要的能力付费。

如果 Claude 已经拥有全部相关上下文、也明显尝试过了，但依然出错，这就是该选用更大模型的信号。如果你正在用更大的模型，而近期的工作一直都是常规性的，切换到更小的模型通常能提升速度、降低成本，同时不影响输出质量。

努力：Claude 没有足够努力地尝试

如果 Claude 出错是因为跳过了某个文件、没有运行测试，或者没有对自己的工作做双重核实，那就选用更高的努力级别。这一点在你选用的努力级别低于该模型默认水平时，尤其相关。

Fable、Opus、Sonnet 的比喻：专才、专家与全才

我喜欢这样理解这两个设置之间的关系：Fable 是一位见过几乎没人见过的问题的专才，Opus 是专家，而 Sonnet 是一位非常出色的全才。努力级别决定的是，这三者中的任何一个会在你的任务上花多少时间。

低努力级别的 Opus，就像是找一位在你这类问题上经验深厚的专家，花五分钟跟你聊聊。他们带来的是那种在你的代码库里哪儿都找不到的知识：见过的各种模式、知道该留意的坑，这种东西只有解决过大量类似问题的人才具备。但只给他们五分钟，意味着他们只能快速扫一眼你的代码，而不是仔细研读。

高努力级别的 Sonnet，就像是给一位非常出色的全才一整个下午的时间。他们会读遍所有内容，跑一遍流程，反复核实自己的工作，最终对你这段具体的代码形成透彻的理解。他们相对欠缺的，是那种"这个问题我以前正好见过"的直觉。

即便是低努力级别的 Fable，也像是那位专才只是瞥了一眼所有人都卡住的问题，却依然能发现别人都没注意到的关键点。这种识别能力，正是你付出最多成本所换来的东西，所以更值得把它留给那些真正需要的任务。

这三者都不是绝对意义上"更好"，模型设置大致对应"能力有多强"，努力设置大致对应"做得有多彻底"。大多数实际任务，两者都需要一些。

努力、模型与 token 消耗

那么模型选择、努力级别和 token 消耗到底是如何相互作用的？这取决于具体任务。

对于常规工作，在相同的努力级别下，两种模型通常都能做对。更大的模型会因为额外的验证步骤而消耗更多 token，而且每个 token 的单价也更高。这就是为什么在常规性的工作阶段切换到更小的模型，能在不影响质量的前提下真正省钱。

对于更难的多步骤工作，情况则不同。较小的模型必须一点点逼近自己能力的极限，消耗大量的迭代次数；而较大的模型则能用更少的步骤达到同样的质量标准。

你为更大的模型支付的单 token 价格更高，但在那些真正能把较小模型逼到极限的任务上，整个任务的总成本反而可能更低。更重要的是，更大的模型能完成一些较小模型无论调到多高努力级别都无法完成的任务。

这一点在 Fable 身上表现得最为明显。在长时间、多步骤的工作中，它遥遥领先。在我们的测试中，它完成了 Opus 和 Sonnet 在任何努力级别下都无法完成的工作。它的单 token 成本也最高，这也是为什么应该把它留给真正需要的工作。

上面这些图表要传达的关键点是：努力级别决定的是 Claude 愿意沿着这条曲线走多远，但这并不意味着 Claude 需要走那么远才能完成任务。

还有一个细微之处：努力级别会影响 token 消耗，但并不会限制它。系统中唯一的硬性上限是 max_tokens，一旦触发，它会在响应流中途直接截断输出。这是一个相当粗暴的手段，主要与 API 开发者相关。相对更柔和的控制手段，比如任务预算（task budgets），或者在提示词中直接要求 Claude 保持简洁，是更有用的工具。它们更像是模型被训练去遵循的指导原则——当接近限制时，它会主动收尾——而不是一堵会直接撞上去的墙。

从默认设置开始，再去调整旋钮

大多数时候，你根本不需要去想这两个设置中的任何一个。当结果不如预期时，问问自己："是 Claude 不够懂，还是它没有足够努力地尝试？"然后据此进行调整。
