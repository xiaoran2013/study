循环入门：了解 Claude Code 团队定义的智能体循环

原文标题：Getting started with loops ｜ 发布日期：2026 年 6 月 30 日 ｜ 分类：Claude Code ｜ 产品：Claude Code ｜ 阅读时长：约 5 分钟 ｜ 查看原文

了解 Claude Code 团队如何定义智能体循环，本文提供从基于轮次、基于目标、基于时间到主动式循环的实用进阶指南，并说明各类循环分别适合什么时候使用。

目前有很多人在讨论"设计循环"，而不是"给编程智能体写提示词"。如果你花点时间在 X（推特）上想搞清楚"循环"到底是什么，你会看到五花八门的答案。

在 Claude Code 团队内部，我们把循环定义为：智能体不断重复工作周期，直到满足某个停止条件为止。我们根据以下几点，把循环划分为几种不同类型：

  • 它们是如何被触发的
  • 它们是如何被停止的
  • 使用了哪种 Claude Code 原语（primitive）
  • 每一种最适合哪类任务

我们会介绍主要的循环类型、各自适用的时机，以及如何在控制 token 用量的同时保持代码质量。并不是所有任务都需要复杂的循环；从最简单的方案开始，有选择地使用这些模式。

基于轮次的循环（Turn-based loops）

  • 触发方式： 用户发出的提示词。
  • 停止条件： Claude 判断自己已经完成任务，或者需要更多上下文。
  • 最适合： 不属于常规流程或日程安排的、较短的任务。
  • 用量管理方式： 写具体的提示词，并借助技能改进验证环节，以减少轮次数量。

你发送的每一条提示词，都会启动一个由你手动主导每一轮的循环。Claude 收集上下文、采取行动、检查自己的工作，如有需要就重复这一过程，然后给出回应。我们把这称为"智能体循环"。

举例来说，让 Claude 创建一个点赞按钮。它会读取你的代码，做出修改，运行测试，然后把它相信可以工作的东西交还给你。接着你需要手动检查这份工作，再写下一条提示词。

你可以把自己手动执行的步骤编写成一份 SKILL.md，从而改进验证环节，让 Claude 能够端到端地检查更多自己的工作。这应该包含能让 Claude 看到、度量或操作结果的工具或连接器。检查项越是可量化，Claude 就越容易做到自我验证。

举例来说，你可能会在 SKILL.md 文件里这样写：

---
name: verify-frontend-change
description: Verify any UI change end-to-end before declaring it done.
---

# Verifying frontend changes
Never report a UI change as complete based on a successful edit alone. Verify it the way a human reviewer would:

1. Start the dev server and open the edited page in the browser.
2. Interact with the change directly. For a new control (button, input, toggle): click it, confirm the expected state change, and screenshot before/after.
3. Check the browser console: zero new errors or warnings.
4. Use the Chrome Devtools MCP, run a performance trace and audit Core Web Vitals.

If any step fails, fix the issue and rerun from step 1 — do not hand back partially verified work.

基于目标的循环（/goal）

  • 触发方式： 实时的手动提示词。
  • 停止条件： 达成目标，或者达到最大轮次数。
  • 最适合： 拥有可验证退出条件的任务。
  • 用量管理方式： 设定具体的完成标准和明确的轮次上限，比如"尝试 5 次后停止"。

有时候，一轮是不够的，尤其是面对更复杂的任务时。智能体在可以迭代的情况下表现会更好。你可以通过 /goal 明确定义"完成"是什么样子，来延长 Claude 持续迭代的时长。

当你定义好成功标准后，Claude 就不必自己判断什么叫"足够好"、从而提前结束循环。每次 Claude 想要停下来时，一个评估模型都会检查你设定的条件，如果目标尚未达成，或者还没到达你设定的轮次上限，就会让它继续工作。

这正是为什么确定性的标准——比如通过测试的数量，或者达到某个分数门槛——会如此有效。

例如：

/goal get the homepage Lighthouse score to 90 or above, stop after 5 tries.

（把首页的 Lighthouse 分数提高到 90 分以上，尝试 5 次后停止。）

基于时间的循环（/loop 与 /schedule）

  • 触发方式： 指定的时间间隔。
  • 停止条件： 你取消它，或者工作完成（PR 被合并、队列清空）。
  • 最适合： 周期性重复的工作，或者与外部环境/系统对接的工作。
  • 用量管理方式： 设置更长的时间间隔，或者根据事件而不是时间来做出反应。

有些智能体工作是周期性重复的：任务本身不变，只有输入在变化。比如，每天早上总结一遍 Slack 消息。还有一些工作依赖外部系统，一种简单的对接方式，就是按固定间隔去检查它，并对变化做出反应。比如一个可能会收到代码评审或者 CI 失败的 PR。

对于这类情况，你可以用 /loop 来触发 Claude 的运行，它会按固定间隔重新运行同一条提示词。例如：

/loop 5m check my PR, address review comments, and fix failing CI

（每 5 分钟检查我的 PR，处理评审意见，并修复失败的 CI。）

/loop 运行在你的电脑上，所以一旦你关掉电脑，它就会停止。你可以通过 /schedule 创建一个例行任务（routine），把循环迁移到云端运行。

主动式循环（Proactive loops）

  • 触发方式： 事件或日程安排，实时过程中没有人工参与。
  • 停止条件： 每个任务在达成目标后退出。例行任务本身会一直运行，直到你把它关闭。
  • 最适合： 持续产生的、定义明确的重复性工作：bug 报告、问题分诊、迁移、依赖升级等等。
  • 用量管理方式： 把例行任务路由给更小、更快的模型，把能力最强的模型留给需要判断力的决策。

上面这些原语，再加上 Claude Code 的其他功能，比如自动模式（auto mode）和动态工作流（dynamic workflows，研究预览版），可以组合成一个用于长时间运行工作的循环。

举例来说，要处理不断涌入的反馈，你可以这样组合：

  1. /schedule（研究预览版），运行一个检查新报告的例行任务
  2. /goal 定义"完成"是什么样子，用技能记录如何验证它
  3. 动态工作流，用来编排负责分诊每份报告、修复问题并评审修复结果的多个智能体
  4. 自动模式，让例行任务运行时不需要停下来请求许可

把这些组合起来，一条提示词可能是这样的：

/schedule every hour: check #project-feedback for bug reports. /goal: don't stop until every report found this run is triaged, actioned, and responded to. When fixing a bug, use a workflow to explore three solutions in parallel worktrees and have a judge adversarially review them.

（/schedule 每小时：检查 #project-feedback 频道里的 bug 报告。/goal：在本次运行中发现的每一份报告都被分诊、处理并回复之前，不要停下。修复 bug 时，用一个工作流在三个并行的工作树中探索三种方案，并让一个裁判智能体对它们进行对抗式评审。）

保持代码质量

一个循环产出的质量，取决于围绕它构建的系统。在设计这套系统时：

  • 保持代码库本身的整洁： Claude 会遵循你代码库里已经存在的模式和约定。
  • 给 Claude 一种验证自己工作的方式： 用技能把"好的标准"（对你和你的团队而言）编码下来。
  • 让文档触手可及： 框架和库的文档里有最新的最佳实践。
  • 用第二个智能体做代码评审： 一个带着全新上下文的评审者，偏见更少，也不会受主智能体推理过程的影响。你可以使用内置的 /code-review 技能，或者面向 GitHub 的 Code Review 功能。

当某一次结果没有达到标准时，不要止步于修复这一个具体问题，而应该尝试把它编码进系统里，让所有后续的迭代都能从中受益。

管理 token 用量

为了管理 token 用量，循环应该有清晰的边界：

  • 为任务选择合适的原语和模型： 较小的任务不需要多个智能体或循环。有些任务可以使用更便宜、更快的模型。
  • 定义清晰的成功与停止标准： 明确说明"完成"是什么样子，这样 Claude 就能更快（但不会过早）地得出解决方案。
  • 在大规模运行前先做试点： 动态工作流可能会生成上百个智能体。先在一小部分工作上评估用量。
  • 对确定性的工作使用脚本： 运行一个脚本比让模型逐步推理要便宜得多。比如，一个 PDF 技能可以自带一个填表脚本，让 Claude 每次直接运行它，而不是每次都重新推导代码。
  • 不要让例行任务运行得比实际需要的更频繁： 让检查间隔匹配你所监控对象实际的变化频率。
  • 审查用量： /usage 命令会按技能、子智能体和 MCP 细分最近的用量，不带参数的 /goal 会显示目前为止的轮次数和 token 用量，/workflows 会显示每个智能体的 token 用量，你可以随时停止任何一个智能体。

开始使用

总结如下：

循环类型你交出的部分适用场景使用工具
基于轮次检查权你在探索或做决策自定义验证技能
基于目标停止条件你清楚"完成"是什么样子/goal
基于时间触发时机工作按日程在你的项目之外发生/loop、/schedule
主动式整个提示词工作是重复性的且定义明确的以上所有工具，再加上动态工作流

要开始使用循环，先看看你目前正在做的工作。挑一个你自己是瓶颈的任务，问问自己可以交出去哪一部分：你能不能写出验证检查？目标是否足够清晰？工作是否会按日程到来？

一旦有了想法，就把循环跑起来，观察结果——比如它在哪里卡住、或者做得超出了范围——不要害怕对它进行迭代。
