集中管理 MCP 连接器的授权

原文标题：Centrally manage authorization for MCP connectors ｜ 发布日期：2026 年 6 月 18 日 ｜ 分类：Enterprise AI / Product announcements ｜ 产品：Claude Enterprise、Claude apps ｜ 阅读时长：约 6 分钟 ｜ 查看原文

管理员现在可以通过身份提供商，为整个组织统一开通 MCP 连接器，首先支持 Okta。用户在首次登录时会自动获得连接器访问权限，授权由所在组织集中配置完成。

连接器让 Claude 在工作中更加有用——它们把团队已经在使用的工具中的上下文，带给 Claude。此前，开通连接器需要在两个步骤上分别采取行动：管理员为整个组织启用某个连接器，然后每一位用户还需要自行完成授权。

企业管理式授权简化了第二个步骤。管理员只需一次性完成对某个连接器的授权，用户则通过他们已经拥有的 IdP 组和角色继承访问权限，连接器在用户第一次打开 Claude 时就已经就绪。最终结果是：终端用户实现了零操作的连接器配置。

企业管理式授权是 Model Context Protocol（MCP）"企业管理式授权（Enterprise-Managed Authorization）"扩展的首个实现。它构建在一个开放标准之上，因此任何连接器都可以支持它——包括你自己团队构建的自定义连接器——并且对每一位 Claude 客户而言，它们的运作方式都是一致的。

运作方式

把你的身份提供商连接到 Claude，然后选择要为你的组织启用哪些 MCP 连接器。当一名员工登录时，他们的连接器就已经就绪。在 Claude 聊天、Claude Code 和 Cowork 之间，访问权限保持一致。

对管理员来说，这把 MCP 访问管理纳入了你已经用来管理其余技术栈的同一套工作流程：一次性开通，按组划定范围，通过 IdP 管理撤销权限。由于通过 IdP 检查访问权限几乎没有摩擦成本，管理员可以缩短访问令牌的有效期而不影响生产力——因此当某人被取消授权时，他们的连接器访问权限会迅速失效，而不是滞留在一个旧令牌上。访问权限通过你已经信任的身份提供商来运行，因此连接器和其他一切一样，都受到相同的安全与访问控制约束，而不是一个需要单独监控的独立入口。

管理员还可以要求某个连接器只能通过 IdP 进行连接，这样可以把工作用途和个人用途干净利落地区分开来，防止有人不小心把个人账户连接到工作工具上。

与生态系统共同构建

企业管理式授权涵盖三类角色：负责管理访问权限的身份提供商、支持该标准的 MCP 服务提供商，以及正在为团队部署受管连接的 Claude 客户。

身份提供商。 Okta 在发布时即获得支持，未来还将支持更多身份提供商。

MCP 服务提供商。 Asana、Atlassian、Canva、Figma、Granola、Linear 和 Supabase 在发布时即支持企业管理式授权，Slack 即将支持。

Claude 客户。 Hubspot、Ramp 和 Webflow 等组织，正在为其团队推广企业管理式授权。

开始使用

企业管理式授权目前已面向 Claude Team 和 Enterprise 套餐的客户开放测试版。在我们的帮助中心了解更多信息，并申请加入以开始使用。

任何身份提供商或 MCP 服务提供商都可以通过实现 MCP 授权规范的开放扩展来添加对企业管理式授权的支持。可以提交申请以加入测试版。
