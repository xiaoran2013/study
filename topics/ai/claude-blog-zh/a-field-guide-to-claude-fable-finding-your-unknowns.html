Claude Fable 5 实战指南：找到你的"未知"

原文标题：A field guide to Claude Fable 5: Finding your unknowns ｜ 发布日期：2026 年 7 月 6 日 ｜ 分类：Claude Code ｜ 产品：Claude Code ｜ 阅读时长：约 5 分钟 ｜ 查看原文

在使用 Claude Code 工作时，我常常会想起"地图"和"疆域"之间的差别。

地图，是对要完成的工作的一种呈现，它是我的提示词、技能和上下文——是我给 Claude 的东西。而疆域，则是工作真正需要发生的地方：代码库、真实世界，以及它实际存在的种种约束。

地图和疆域之间的差距，就是我所说的"未知（unknowns）"。当 Claude 遇到一个未知时，它需要根据自己对我想要什么的最佳猜测来做出决定。要完成的工作越多，Claude 可能遇到的未知也就越多。

Claude Fable 是第一个让我发现——工作质量的瓶颈，在于我澄清它的未知的能力——的模型。

需要指出的是，仅仅提前做计划并不总是够用。你可能会在实现过程的深处发现未知，或者你的未知也可能指向一个事实：你其实应该用一种完全不同的方式来解决这个问题。

我发现，和 Fable 一起工作是一个迭代的过程——在实现之前、实现过程中、以及实现之后，不断发现自己的未知。

认识你的未知

你的未知都是什么？当我带着一个问题来找 Claude 时，我倾向于把它拆成四类：

  • 已知的已知（Known Knowns）： 本质上就是我提示词里的内容——我明确告诉智能体我想要什么。
  • 已知的未知（Known Unknowns）： 我还没搞清楚、但我知道自己没搞清楚的部分。
  • 未知的已知（Unknown Knowns）： 那些显而易见到我根本不会写下来、但一旦看见就能一眼认出来的东西。
  • 未知的未知（Unknown Unknowns）： 我完全没有考虑过的东西。我不知道自己不知道的知识。我是否清楚一件事情能做到多好？

最优秀的智能体编程者，未知相对是最少的。看 Boris 或 Jarred 这样的人写提示词，很明显能看出他们对自己想要什么有非常细致的了解。他们和代码库、和模型的行为模式都保持着深度同步。

但他们同样会预设未知的存在。在很多方面，减少并提前规划自己的未知，正是智能体编程的核心技能。而幸运的是，这项技能是可以通过和 Claude 一起工作来不断提升的。

帮助 Claude 帮助你

给 Claude 下指令是一种微妙的平衡。如果你说得太具体，Claude 会照做，即便这时候转向另一个方向可能更合适。如果你说得太模糊，Claude 常常会基于行业最佳实践做出选择和假设，而这些未必适合你的任务。

当你没有考虑到自己的未知时，两种失败都可能发生。你不知道什么时候道路上会布满障碍，也不知道什么时候道路本该畅通、但你其实希望 Claude 主动转向。

Claude 可以帮你更快地发现自己的未知。它能极快地搜索你的代码库和互联网，而且它对大多数话题的了解都远超普通人。它也能更快地从失败中迭代。

这个过程中最重要的一环，是给 Claude 提供关于你出发点的上下文。比如，告诉它你目前的思考进展到了哪一步；坦白你在这个问题和代码库上的经验水平；让它像一个思考伙伴一样和你一起工作。

在这篇文章中，我会详细介绍一些我用来挖掘这些未知的模式，包括：

实现之前：

  • 盲点排查
  • 头脑风暴与原型
  • 访谈
  • 参考资料
  • 实现计划

实现过程中：

  • 实现笔记

实现之后：

  • 说明材料与讲解文档
  • 测验

实现之前

盲点排查

开始一项工作时，你能做的最有用的事情之一，就是了解自己的盲点。比如，如果你正在代码库中一个陌生的部分写功能，或者在用 Claude 帮你处理不熟悉的工作（比如迭代设计），你很可能会有大量的"未知的未知"。

你可能不知道该问什么问题、什么样的结果算是好的、之前已经做过哪些历史工作，或者应该避开哪些坑。

在这种情况下，你可以让 Claude 帮你找出自己的"未知的未知"，并向你解释清楚。我喜欢直接用"盲点排查"和"未知的未知"这两个说法。给它提供关于你是谁、你已经知道什么的背景信息，通常对 Claude 找到与你协作的最佳方式很重要。

示例提示词：

  • "我正在给项目添加一个新的身份认证提供商，但我对这个代码库里的认证模块完全不了解。能不能做一次盲点排查，帮我找出相关的未知的未知，并教我如何更好地向你提问？"
  • "我不知道什么是调色，但我需要给这段视频调色。能不能教我理解自己在调色方面的未知的未知，好让我能更好地向你提问？"

头脑风暴与原型

当我在一个充满大量"未知的已知"的领域工作时——也就是那些我只有看到了才知道该如何定义的标准——我喜欢让 Claude 陪我一起做头脑风暴和原型。

在原型阶段尽早识别并说出这些"未知的已知"，价值极大，因为在实现过程中才发现它们，代价（相对而言）会高得多。一个功能或规格上的微小改动，可能会在代码中导致截然不同的实现，而且智能体想要撤销之前的改动，往往也会更困难。

举个例子，你可能只是想看看在一个界面框架里加一个按钮会是什么样子，而不必去接通后端路由或在前端维护额外的状态。

另一个例子是视觉设计——对我来说，这是一种很难用语言表达、但一看到就知道自己想要什么的东西。在这些情况下，我会要求 Claude 针对一个 artifact 给出几种不同的设计方案。

我也几乎在每一次编程会话开始时，都会先进行一个探索或头脑风暴的阶段。这有助于我从一开始就带着明确的意图去界定项目的范围。Claude 常常能找到我自己会错过的高价值方案，但有时也会因小失大、只见树木不见森林。头脑风暴能防止我把范围设定得太窄或太宽。

示例提示词：

  • "我想为这些数据做一个仪表盘，但我完全没有视觉审美，也不知道有哪些可能性。给我做一个 HTML 页面，展示 4 种风格迥异的设计方向，让我可以对它们做出反应。"
  • "在接通任何东西之前，先做一个单独的 HTML 文件，用假数据模拟新的编辑器工具栏。我想先对布局做出反应，再让你去动真正的应用。"
  • "我大致的问题是这样：用户在完成入职后就流失了。搜索一下代码库，头脑风暴出 10 个我们可以介入的地方，从成本最低到最有野心的排列。我会告诉你哪些方案打动了我。"

访谈

在做了足够的头脑风暴之后，我可能依然还有未知。

这种情况下，我会让 Claude 就任何未知或含糊之处对我进行访谈。在让 Claude 访谈你的时候，尽量给它一些关于你的问题的背景信息，以引导它的提问方向。

示例提示词：

  • "针对任何含糊不清的地方，一次问我一个问题，优先问那些一旦我回答就会改变架构的问题。"

参考资料

有时候你无法详细描述自己想要的东西。比如你可能没有相应的语言来表达，或者它复杂到需要花很长时间才能说清楚。

在这种情况下，最好的办法是提供一份参考资料。虽然你可以附上示意图、文档或图片，但最好的参考资料，绝对是源代码本身。

如果你有一个库以某种特定方式实现了某个功能，或者你非常喜欢某个设计组件，那就直接把 Fable 指向那个文件夹，告诉它该关注什么，即使它是用不同的语言写的也没关系。相比截图这类材料，这能为 Claude 提供更丰富的标记和结构细节。

示例提示词：

  • "vendor/rate-limiter 里的这个 Rust crate 实现了我想要的那种退避（backoff）行为。读一下它，然后在我们的 TypeScript API 客户端里用同样的语义重新实现一遍。"

实现计划

当我觉得自己准备好开始实现时，我倾向于让 Claude 整理出一份实现计划供我审阅。这份计划会聚焦于最可能发生变化的部分，比如数据模型、类型接口，或者用户体验流程。这样一来，Claude 就能把我实际可能需要调整的地方提前浮现出来。

示例提示词：

  • "用 HTML 写一份实现计划，但要以我最可能想要调整的决策打头：数据模型的变化、新的类型接口，以及任何面向用户的部分。把机械性的重构内容放在最后，那部分我信得过你。"

实现过程中

实现笔记

一旦我对计划感到满意，我会开一个新会话，把所有相关的 artifacts 传给提示词。这样能给 Claude 一个全新的上下文窗口，同时保留它在规划阶段汇总的全部信息。比如，我可能会传入一份规格文件和一个原型，然后让智能体去实现它。

但事实是，无论你做了多少规划，总有"未知的未知"潜伏在某处。智能体可能会在工作过程中发现，由于代码中出现了某个边缘情况，它需要改变策略。

我会让 Claude Code 保留一份临时的 "implementation-notes.md"（或 .html）文件，用来记录它做出的各项决策，以便我们在下一次尝试时能有所借鉴。

示例提示词：

  • "维护一份 implementation-notes.md 文件。如果你遇到一个迫使你偏离计划的边缘情况，选择保守的方案，把它记录在'偏离项（Deviations）'下，然后继续推进。"

实现之后

说明材料与讲解文档

交付一项工作最重要的环节之一，就是获得他人的认同和审批。在最终文档中构建说明材料和讲解性 artifacts，有助于：

  • 当审阅者一开始也和你一样存在同样的未知时，加快他们理解的速度
  • 当专家希望看到你已经考虑到他们本会预见到的未知和常见失败点时，加快审批速度

示例提示词：

  • "把原型、规格说明和实现笔记打包成一份文档，我可以直接丢进 Slack 里去争取大家的认同。开头放上演示 GIF。"

测验

经过一场漫长的工作会话之后，Claude 可能完成的工作，远比我意识到的要多。仅仅阅读代码差异（diff），只能让我对发生了什么有一个粗浅的理解，因为很多行为都取决于既有的代码路径。

在获得大量背景信息之后，让 Claude 就这次改动对我进行测验，能帮助我理解到底发生了什么。只有在我完美通过测验之后，我才会合并代码。

示例提示词：

  • "我想确保自己完全理解这次改动中发生的一切。给我一份关于这些改动的 HTML 报告，让我阅读并理解上下文、直觉、做了什么等等，并在最后附上一个我必须通过的关于这些改动的测验。"

这一切如何汇聚：Fable 的发布

Fable 的发布视频从头到尾都是用 Claude Code 剪辑的。这对我来说是一个全新的领域，我在这方面绝不是专家。

所以我从自己确实知道的东西开始入手。我知道 Claude 可以用代码来剪辑视频和做转录，但我不确定它是否足够准确。于是我让 Claude 向我解释像 Whisper 这样的转录工具是如何工作的，以及我是否能用 ffmpeg 准确剪掉像"嗯""啊"这样的语气词或长时间的停顿。

我希望 Claude 打造一个能和我说话的字词精确同步的界面，但不确定这是否可行，于是我让 Claude 用 Remotion 和一份转录稿做一个原型视频，看看是否行得通。

最后，视频本身看起来色调有点发闷，我知道这是调色的结果，但我其实并不真正了解什么是调色。我第一次尝试是让 Claude 做几个不同版本供我挑选，但我意识到，我压根不知道调色调到"好"是什么样子。于是，我转而让 Claude 教我了解调色，来发现我自己的未知。

让地图匹配疆域

模型能力越强，只要方法得当，你能实现的目标就越多。当一个长周期任务的结果出错时，很可能说明你需要花更多时间去界定自己的未知，或者制定一份能让你和 Claude 在过程中不断适应调整的实现计划。

每一份讲解文档、每一次头脑风暴、每一次访谈、每一个原型、每一份参考资料，都是一种低成本的方式，能让你在"修复代价变得高昂"之前，先弄清楚自己原本不知道的事情。

所以，从让 Claude 帮你找到自己的未知开始，去启动你的下一个项目吧。
