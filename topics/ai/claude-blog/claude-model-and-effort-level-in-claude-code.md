# 选择 Claude Code 中的模型与努力级别

## 文章在讲什么

这篇文章出自 Claude Code 团队的技术成员 Lydia Hallie，主题是帮助用户理解 Claude Code 中两个最核心的设置：**模型选择（model selection）**和**努力级别（effort level）**。

很多用户以为"更大的模型 = 更好的答案"和"更高的努力级别 = 更长的思考时间"。第一个假设基本正确，但第二个理解是严重简化。作者指出，努力级别实际上控制的是 Claude 在一个任务上**做多少工作**，包括读多少文件、做多少验证、走多少步骤——而不仅仅是"想了多久"。

这篇文章的背景是：Claude Code 是 Anthropic 推出的命令行编程工具，用户需要在这两个维度上做选择，但很多人并不理解它们真正在控制什么，导致要么选错设置浪费成本，要么在问题出现时不知道该调哪个参数。

## 核心概念讲解

### 模型选择：选择哪套"权重"

当你选择 Sonnet、Opus 或 Fable 时，你实际上是在选择**不同的权重集合（weights）**来处理你的请求。

什么是权重？作者用了一个很清晰的解释：

> 模型接收一段文字后，要预测下一个 token 是什么。它通过把输入经过一套大型矩阵（ billions of numbers organized into large matrices）来计算每个 token 的概率。这套矩阵中的数字就是权重。

关键理解：

- **权重是固定的**。训练完成后，权重就冻结了。你的提示词、CLAUDE.md、上下文都不会改变权重。这叫做"推理（inference）"——用训练好的模型，权重不变。
- **模型的知识存在于权重中**。如果一个库在模型训练时还不存在，它就不在权重里。你可以把文档放进上下文让 Claude 参考，那是"引导（steering）"而非"教学（teaching）"。
- **换模型 = 换权重**。更大的模型意味着更强大的能力，但也意味着更高的 token 成本。

### 努力级别：控制做多少工作

努力级别（effort）不是简单的"思考时间"，它控制的是 Claude **为一个请求做多少工作**，包括：

- 读多少文件
- 做多少验证
- 在中途停下来问你之前，走多远

当努力级别高时，Claude 会先制定计划，并根据执行结果动态调整——如果第一步就找到 bug，后面的检查可能就跳过了。作者特别强调，训练过程中团队会注意防止"过度思考（overthinking）"，因为那会降低效率。

### 两者之间的关系

作者的比喻非常到位：

> Fable 是 specialist（专家），见过几乎没人见过的问题；Opus 是 expert（资深专家）；Sonnet 是很棒的 generalist（通才）。努力级别决定这三者在你任务上花多少时间。

- **Opus 低努力**：像给一位资深专家 5 分钟，他有深度经验，但只能快速浏览你的代码。
- **Sonnet 高努力**：像给一位很棒的通才整个下午，他会读所有内容、运行测试、仔细检查，但对"我见过完全一样的问题"这种识别能力较弱。
- **Fable 低努力**：即便只给片刻，这位专家也能看到别人看不到的关键点——这就是你花最多钱买的东西。

## 文章主体讲解

### 第一部分：纠正两个误解

这一节作者指出用户的两个常见假设，然后说明哪个对、哪个错。

**关键点**：

- "更大的模型 = 更好的输出"——这个假设**正确**，因为最大的模型在行业基准测试中确实更强。
- "更高的努力级别 = 更长的思考时间"——这个假设**过于简化**。努力级别控制的是整体工作量，包括文件读取、验证、步数等，而不仅仅是思考时间。

### 第二部分：模型选择的技术原理

这一节深入解释模型实际上是什么。

**为什么重要**：理解模型选择的本质，才能明白为什么换模型不等于"给模型灌输知识"。

**关键点**：

- 模型的工作是预测下一个 token（词元）。它对词汇表中每个 token 计算概率，然后从最高概率中选取。
- 权重（weights）是数十亿个数字，组织成大型矩阵。输入经过这些矩阵的乘法运算，得出概率分布。
- 权重在训练时确定，之后完全固定。你的任何输入都无法改变权重。
- 幻觉（hallucination）不是因为"查不到"，而是权重根据训练时学到的模式生成了看起来合理的 token 序列。
- 换模型就是换一套权重，同时也决定了每个输出 token 的成本。

### 第三部分：努力级别如何工作

这一节解释 effort 设置如何影响 Claude 的行为。

**为什么重要**：理解 effort 的真正含义，才能在结果不好时判断是否需要调整它。

**关键点**：

- Claude 生成的 token 分三类：**思考（thinking）**、**工具调用（tool calls）**、**给你的文字（text to you）**。它们都是普通输出 token，费率相同。
- 思考 token 和你读到的文件一样，都会留在上下文中影响后续推理。
- effort 作为请求的一部分发送给模型，模型根据它决定行为：需要多深入、多确定才能认为任务完成。
- effort 在每个回合都会重新评估。计划不是固定的，Claude 会根据执行结果动态调整。
- 高 effort 意味着更倾向于验证和双重检查，但不会为简单任务人为增加使用量。

### 第四部分：如何选择努力级别

这一节给出实操建议。

**为什么重要**：用户不应该每个任务都纠结于 effort 设置。

**关键点**：

- 大多数任务应该使用**模型的默认努力级别**，这是模型认为大多数人愿意为一个任务花费的工作量。
- effort 是"手动覆盖"，用于你有强烈偏好（更彻底或更快）时。
- 应该把它当作**整体偏好**，而不是逐任务决策。
- 一个新洞察：Opus 4.8 使用默认 effort 时，相比 Opus 4.7 的默认 effort，能产生更好结果且 token 数量相近。

### 第五部分：Claude 犯错时该改什么

这一节给出核心决策框架。

**为什么重要**：这是全文最实用的部分，帮助用户在结果不满意时做出正确调整。

**关键点**：

- 首先检查上下文：提示是否太模糊？连接的工具是否正确？技能是否配备？
- 如果上下文清晰但仍然出错，问自己：**是尝试得不够，还是知道得不够？**

| 情况 | 应该调整什么 | 例子 |
|------|-------------|------|
| 问题太难 | 选更大的模型 | 微妙的 bug、不熟悉的领域、架构决策 |
| Claude 跳过文件、没运行测试、没检查 | 提高 effort 级别 | 你选了低于默认的 effort |

**模型大小的建议**：

- 更大的模型更适合处理模糊性；更小的模型在精确指令下表现更好。
- 如果任务一直是 routine 的，可以降级模型以提升速度并降低成本。
- Fable 的独特价值：即便低 effort，它也能完成 Opus 和 Sonnet 在任何 effort 下都无法完成的任务。

### 第六部分：成本与 token 消耗

这一节分析两个设置如何共同影响费用。

**为什么重要**：帮助用户在质量和成本之间做出明智权衡。

**关键点**：

- **常规任务**：两个模型在同一 effort 下通常都能完成，提高 effort 不值得。小模型成本更低。
- **困难的多步任务**：小模型需要"苦熬"，烧掉大量迭代；大模型用更少的步骤达到相同质量。总成本可能反而更低。
- **Fable 的特殊地位**：在长多步任务上遥遥领先，能完成其他模型任何 effort 下都无法完成的工作。这也是它最贵的原因——应该留给真正需要它的任务。
- effort 不限制 token 消耗，只是影响消耗倾向。硬上限是 max_tokens（会截断），更柔和的控制如"任务预算"或"请保持简洁"更实用。

### 第七部分：从默认值开始，然后调旋钮

这是全文的收尾，与开头呼应。

**关键点**：

- 大多数时候不需要考虑这两个设置。
- 结果不理想时问："是不够尽力还是不够有知识？"然后相应调整。
- 作者的核心建议始终是：**从默认开始，按需微调**。

## 关键对比表

### 模型对比

| 特性 | Sonnet | Opus | Fable |
|------|--------|------|-------|
| 定位 | 很棒的通才（generalist） | 资深专家（expert） | 专家（specialist） |
| 擅长 | 精确指令下的常规任务 | 深度经验、复杂任务 | 解决别人解决不了的问题 |
| 性价比 | 常规任务成本低 | 中等 | 只留给需要它的任务 |
| 特殊能力 | — | — | 能完成其他模型任何 effort 下都无法完成的工作 |

### 模型 vs 努力级别

| 维度 | 模型选择 | 努力级别 |
|------|---------|---------|
| 控制什么 | 能力范围（权重） | 工作量（文件、验证、步数） |
| 何时调高 | 问题太难、知识不够 | Claude 跳过步骤、没充分验证 |
| 类比 | 选谁来做 | 给这个人多少时间 |
| 成本影响 | 每个 token 的单价 | 生成的 token 总数 |

### 何时改什么

| 错误表现 | 原因判断 | 调整动作 |
|---------|---------|---------|
| 自信地调用不存在的 API（幻觉） | 知识不够（模型没学过） | 换更大模型或提供上下文引导 |
| 跳过文件、没运行测试 | 尝试不够（effort 太低） | 提高 effort 级别 |
| 任务一直 routine，质量没问题 | 能力过剩 | 降级模型省成本 |
| context 清晰但仍然出错 | 知识不够 | 换更大模型 |

## 关键句短摘译

**1. "Effort level controls how much work Claude does on your request overall."**
努力级别控制 Claude 在你的请求上做多少工作。

*为什么重要*：这是全文的核心纠正——effort 不只是思考时间，而是整体工作量。

**2. "The weights are where everything the model 'knows' lives."**
权重是模型"知道"的一切所在的地方。

*为什么重要*：帮助读者理解为什么换模型是换能力，为什么上下文不能改变模型的基础知识。

**3. "If a library didn't exist when the model was trained, it isn't in the weights."**
如果一个库在模型训练时还不存在，它就不在权重里。

*为什么重要*：澄清"引导"和"教学"的区别——你放文档进去是引导，不会让模型真的学会那个库。

**4. "Did it not try hard enough, or did it not know enough?"**
是尝试得不够，还是知道得不够？

*为什么重要*：这是作者给出的核心决策框架，也是全文最实用的建议。

**5. "Fable, even at low effort, is that specialist glancing at the problem everyone else is stuck on and still spotting the thing no one else would."**
Fable 即便在低努力下，也是那个瞥一眼所有人卡住的问题、仍然能发现别人发现不了的东西的专家。

*为什么重要*：用生动比喻说明 Fable 的独特价值——买的是那种"识别能力"，不是努力。

**6. "The model setting is roughly how capable; the effort setting is roughly how thorough."**
模型设置大致决定能力；努力级别大致决定彻底程度。

*为什么重要*：两个维度最简单的概括，帮助读者建立直觉。

**7. "Start with the defaults, then reach for the dials."**
从默认值开始，然后再去调旋钮。

*为什么重要*：作者的核心建议，避免过度优化。

## 术语表

| 英文术语 | 中文解释 |
|---------|---------|
| **Model selection** | 模型选择。指在 Claude Code 中选择使用哪个 Claude 模型（Sonnet、Opus、Fable）。 |
| **Effort level** | 努力级别。控制 Claude 在任务上投入多少工作的设置。 |
| **Weights** | 权重。也称 parameters，是数十亿个数字，组织成大型矩阵，决定模型的输出概率。 |
| **Tokens** | 词元。文本处理的基本单位，模型一次生成一个 token。 |
| **Tokenization** | 分词。将文本分割成 token 并映射为整数的过程。 |
| **Inference** | 推理。使用已训练好的模型（权重固定）进行预测的过程。 |
| **Hallucination** | 幻觉。模型生成看似合理但实际错误的输出，因权重中不存在对应知识。 |
| **Steering** | 引导。通过上下文让模型参考特定信息，但不改变模型权重。 |
| **Max_tokens** | 最大 token 数。硬性上限，触发时会截断响应。 |
| **Overthinking** | 过度思考。模型为简单任务花费过多工作，降低效率。 |

## 文章结尾怎么收束

作者在结尾回到了开头的核心信息：**大多数时候你不应该考虑这两个设置**。

全文的逻辑线是：

1. **破除误解**：effort 不只是思考时间。
2. **解释原理**：模型是权重，努力是工作量。
3. **给出框架**：从默认值开始，出问题时问"尽力还是尽力不够"和"够不够知道"。
4. **成本考量**：解释模型和 effort 如何共同影响 token 消耗。
5. **回归默认**：强调不要过度优化，从默认值开始，按需微调。

结尾的潜台词很清楚：**这两个设置是工具，不是目标**。理解它们是为了在真正需要时能做出正确选择，而不是每天纠结于调参。把注意力放回你真正要做的工作上。
