# 为管理员提供更高的可见性和 Claude 消费控制能力

## 文章在讲什么

这篇文章是 Anthropic 官方发布的**产品功能更新公告**，核心主题是：为 Claude Enterprise 的管理员（Admin）提供更强大的**用量可见性**和**成本控制能力**。

**背景是什么？** 随着 Claude 从简单的聊天工具演变为承担复杂代理任务（agentic work）的平台，用量模式和成本结构发生了根本变化——不再是"每人每天问几个问题"那么简单，而是涉及代码生成、文件编辑、API 调用、团队协作等多维度消耗。

**作者想解决什么问题？** 管理员（尤其是 IT 和财务人员）在管理企业级 AI 工具时面临三个核心痛点：

1. **看不见**：不知道团队怎么用、用多少、值不值
2. **控不住**：缺乏灵活的分级控制手段
3. **措手不及**：费用超支后才发现问题，没有预警

这篇文章就是围绕这三个问题，介绍一系列新推出的管理功能。

## 核心概念讲解

在深入正文之前，我们需要理解几个贯穿全文的核心概念：

### 1. Admin Console（管理控制台）

这是管理员登录 Claude Enterprise 后访问的专属界面，用于配置组织设置、查看分析数据、管理用户权限。所有新功能都集成在这里。

### 2. 模型默认设置与权限控制（Model Defaults & Entitlements）

"Entitlements" 在企业软件中指"授权"或"权限配额"。这篇文章里指的是：管理员可以决定哪些**角色（roles）**或**组织层级**可以使用哪些 Claude 模型。

打个比方：初级员工默认使用轻量级模型完成简单任务，资深工程师才能调用最强大的模型处理复杂任务。这是一种**按角色分配资源**的策略。

### 3. Analytics API

允许程序化获取用量数据的接口。管理员可以把 Claude 的使用数据导出到外部系统（如 Datadog、CloudZero），与企业其他云成本一起统一管理。

### 4. Spend Cap（消费上限）

文章提到 Anthropic 已有"各层级的消费上限"功能。这类似于手机套餐的流量封顶——达到上限后服务受限或暂停。

### 5. Skills 和 Connectors

这是 Claude Code 的扩展机制：

- **Skills**：预定义的技能包，增强 Claude 在特定任务上的能力
- **Connectors**：连接外部数据源或工具的桥梁（如连接企业 MCP 服务器）

这两个概念会在"追踪采用情况"部分频繁出现，因为管理员需要了解团队使用了哪些扩展功能。

## 文章主体讲解

### 开篇：为什么需要这些更新

文章开头指出，随着 Claude 承担越来越多"代理性质的工作"（agentic work），用量和成本模式与标准聊天工具截然不同。

**这句话的含义是什么？**

传统的 AI 对话工具使用模式简单：用户问一句，AI 答一句，成本可预测。但代理任务不同——Claude 可能需要：

- 多次调用模型完成一个复杂任务
- 生成并保存多个制品（artifacts）
- 编辑代码文件
- 调用外部工具和 API

这些行为叠加起来，成本增长是非线性的。管理员如果没有合适的工具，很容易在月底看到账单时"大吃一惊"。

作者用"We’re introducing"引出三大类更新：

1. 更丰富的分析功能
2. 模型级别的权限控制
3. 消费预警机制

**注意**：作者强调这些功能是"building on controls Anthropic already provides"——意思是这不是从零开始，而是在已有的控制能力基础上扩展。

---

### 第一部分：追踪采用情况和成本

这一部分的核心是**可见性**——让管理员"看得见"团队怎么用 Claude。

#### 1. 管理员分析面板升级

**讲什么**：原有的分析面板增加了按**组（group）**和**用户（user）**分解的视图，每个用户的用量旁边直接显示对应的成本。

**为什么重要**：这是最基本但最关键的改进。管理员终于可以回答"哪个团队用得最多、花得最多、产出了什么"这类问题。

**关键点**：

- 视图可以直接看到 **artifacts 创建数量**、**文件编辑次数**、**使用的 Skills 和 Connectors**
- 按 **SCIM 组**过滤——SCIM 是企业身份管理系统（如 Okta、Azure AD）的标准协议，管理员可以用 IT 团队已有的组织结构来查看用量，不需要重新建组

#### 2. Claude Code 的两个新标签页

**讲什么**：Claude Code（面向开发者的 CLI 工具）的管理界面新增 **Usage** 和 **Value** 两个标签页。

**为什么重要**：开发者是 Claude Enterprise 的核心用户群体之一，代码工具的使用数据需要专门处理。

**关键点**：

| 标签页 | 内容 |
|--------|------|
| Usage | 活跃开发者数量、会话次数、常用命令排行榜，数据每日更新 |
| Value | 用量与成本汇总，估算生产力提升、每次提交成本、年度价值。公式公开且参数可调 |

**Value 标签页的设计意图**：让管理员不只看到"花了多少钱"，还能量化"带来了多少价值"——这是向管理层汇报的重要素材。

#### 3. Analytics Chat（分析对话）

**讲什么**：管理员可以用自然语言提问，Claude 直接生成图表回答。

**为什么重要**：传统 BI 工具需要写 SQL 或拖拽配置，门槛高。Analytics Chat 把这一过程简化成"问问题"。

**关键点**：

- 示例问题："Which teams doubled their Claude usage this month?"（哪些团队本月用量翻倍了？）
- 示例问题："Where are we getting the most value per seat?"（哪个座位的单位价值最高？）
- 生成的图表可导出分享给利益相关方

#### 4. Analytics API

**讲什么**：提供程序化获取数据的接口，支持外部系统集成。

**为什么重要**：企业的财务和 IT 团队通常不使用管理控制台，而是在自己的系统（Datadog、CloudZero 等）中追踪成本。Analytics API 让数据"流"到他们熟悉的地方。

**关键点**：

- 支持按日期范围、团队、产品、模型过滤
- Skills 报告自己的用量和成本
- 新增端点追踪插件使用和制品创建

#### 5. 个人用量视图

**讲什么**：管理员可以给单个用户开通详细用量数据的访问权限，用户自己也能看到。

**为什么重要**：防止"惊讶式封停"——用户不知道自己的用量快到上限了，突然被中断会很恼火。让用户看到自己的使用趋势，他们会更主动地调整行为。

---

### 第二部分：管理消费的控制功能

这一部分的核心是**控制**——让管理员不只能"看见"，还能"动手"。

#### 1. 模型默认设置与权限

**讲什么**：管理员可以设定新对话的默认模型，并控制哪些角色可以使用哪些模型。

**为什么重要**：不同模型的定价差异很大。如果简单任务默认使用最贵的模型，成本会快速膨胀。合理的默认设置能让日常任务使用性价比更高的模型，复杂任务才调用高级模型。

**关键点**：

- 设置影响 chat、Cowork、Claude Code 三个入口
- 可以按角色或全组织设置
- 目的不是限制，而是引导更合理的资源使用

#### 2. 消费阈值预警

**讲什么**：在消费达到上限的 75% 和 90% 时通知管理员；在 75% 和 95% 时通知用户。

**为什么重要**：这是一道"提前量"机制。管理员不会等到 100% 封顶才措手不及，用户也不会在关键任务进行到一半时被强制中断。

**关键点**：

- 管理员收到预警后可以提前提高上限
- 用户在 75% 时收到通知，95% 时再次通知并可以**直接在应用内申请提高限额**，无需切换工具找管理员

#### 3. Admin API

**讲什么**：允许管理员把成本控制工作流写成脚本，批量执行。

**为什么重要**：对于大型组织，手动管理每个团队的配额不现实。Admin API 让自动化成为可能。

**关键点**：

- 自动审核限额提升申请
- 识别接近限额的成员
- 标记用量快速变化的异常情况

### Getting Started：收尾指引

文章结尾给出了具体的操作步骤，引导管理员：

1. 在管理控制台查看用量和成本分解
2. 按组设置模型默认和消费限额
3. 配置消费阈值预警
4. 使用 Analytics API 将数据拉入现有报表系统

这是一个典型的产品公告结尾结构：**用具体的"怎么做"把读者从"理解了"推向"去试试"**。

   *在达到 75% 和 90% 时向管理员发送预警，让他们在任何人被任务中途阻断之前有时间提高上限。*

   **重要性**：说明预警机制的核心价值——提前干预，避免中断。

4. **"Cost visibility isn't a once-a-month exercise."**

   *成本可见性不是每月一次的事情。*

5. **"He's asking for ROI."**

   *他（CFO）要的是投资回报率。*

   **重要性**：Carter Busse 的话揭示了企业采购 AI 工具的真实驱动力——不是"省钱"，而是"值回票价"。

   *仅看 Token 用量说明不了什么。我真正想看到的是哪些 Skills 被反复使用。*

   **重要性**： Ciro Yamada 指出衡量价值的关键指标——重复使用率，而非绝对消耗量。
