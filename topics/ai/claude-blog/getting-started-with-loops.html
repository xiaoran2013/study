核心概念讲解

什么是循环（Loop）？

文章给循环下了一个定义：

循环 = 代理重复工作周期，直到满足停止条件为止。

这个定义里有三个关键词需要拆开理解：

  • 代理（Agent）：在 Claude Code 里指的是 AI 模型在执行任务时的角色，它可以接收指令、执行动作、返回结果。
  • 重复工作周期：代理不是执行一次就结束，而是会多轮执行，每轮可能做不同的事情。
  • 停止条件：循环不会无限进行，必须有一个明确的退出点。

类比一下：就像你让一个实习生帮你处理文件，你会告诉他"做完这 100 份就停下"或者"处理到没有新文件为止"。循环就是把这个授权过程自动化。

循环的四个维度

文章说，区分不同循环类型要看这四个方面：

维度含义
触发方式（How triggered）循环在什么情况下开始
停止方式（How stopped）循环在什么情况下结束
使用的原语（Claude Code primitive）具体用哪个命令或工具来实现
适合的任务类型哪种工作最适合这种循环

理解这四个维度很重要，因为它们决定了你在循环中需要"交出什么控制权"。文章后面就是围绕这四个维度展开的。

核心心智模型："你交出什么？"

作者在四种循环的对比中，用了一个贯穿始终的问题："你可以把哪一部分工作交给代理？"

  • Turn-based：你把检查工作交出去
  • Goal-based：你把目标定义交出去
  • Time-based：你把触发时机交出去
  • Proactive：你把整个提示交出去

这个思考框架帮助你在实际工作中快速判断该用哪种循环。

四种循环类型

文章用一张表格介绍了四种循环。为了更好地理解，我逐个展开：

1. Turn-based loops（基于轮次的循环）

  • 触发方式：你主动发起下一轮
  • 停止方式：你判断"可以了"才停下
  • 使用场景：探索性任务、不确定要做什么、需要做决策
  • 对应工具：Custom verification skills（自定义验证技能）

这种循环最像我们日常使用 AI 的方式：你问一句，AI 答一句，你觉得满意了就说"可以了"。区别在于，这里 AI 可以多轮执行，而且每次执行后会等待你检查。

适合的场景是当你还在探索问题或者需要判断的时候。比如你不确定代码改对了没有，需要人工介入确认。

2. Goal-based loops（基于目标的循环）

  • 触发方式：代理自动继续
  • 停止方式：达成目标后自动停止
  • 使用场景：目标明确、知道"完成"长什么样
  • 对应工具：/goal 命令

这是最符合直觉的循环类型：你告诉 AI "把这个功能写完"，AI 会一直工作直到他认为完成了。你不需要逐轮检查，只需要定义好目标即可。

关键是你必须有清晰的目标。如果目标模糊，AI 可能无法正确判断何时停止。

3. Time-based loops（基于时间的循环）

  • 触发方式：定时触发（比如每天早上、每小时）
  • 停止方式：按计划停止
  • 使用场景：外部工作、定时任务
  • 对应工具：/loop、/schedule

这种循环的特点是工作发生在你的项目外部，并且按时间表运行。比如定期拉取数据、定期发送报告等。

它的核心价值是让你从"记得去做"这件事中解放出来，交给定时器触发。

4. Proactive loops（主动式循环）

  • 触发方式：AI 根据情况自己决定
  • 停止方式：任务完成或条件不满足
  • 使用场景：重复性且定义明确的任务
  • 对应工具：以上所有，以及 dynamic workflows（动态工作流）

这是最"智能"的循环。你把完整的提示交出去，AI 会自主判断什么时候该做什么。它适用于 recurring（周期性发生）且 well-defined（定义清晰）的工作。

作者把这种放在最后，可能暗示它是最复杂的类型，需要对前三种有充分理解后再使用。

如何开始：行动指南

这一节讲什么：作者给出了从零开始尝试循环的具体步骤。

为什么重要：理论和实践之间有鸿沟，这一段把"知道循环是什么"变成"真的用起来"。

关键点：

  1. 从你现有的工作入手：不要凭空想象一个场景，找一个你现在就在做、且你是瓶颈的任务。
  2. 问自己一个关键问题：哪部分工作我可以交出去？是可以写验证检查？还是目标已经够清晰？还是工作会按时到来？
  3. 先跑起来：运行循环，观察结果，注意它在哪里卡住或者超出范围。
  4. 迭代：不要期望一次就完美，根据观察调整。

这个方法论很务实——从小处着手，观察反馈，逐步完善。它的本质是把循环设计当成一个实验，而不是一个计划。
