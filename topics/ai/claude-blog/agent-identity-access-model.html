文章在讲什么

这篇文章介绍的是 Claude Tag 产品中一种全新的访问控制模型——agent identity（智能体身份）。作者解释了为什么传统的"以用户身份操作"模式无法适用于多人协作的 AI 环境，并详细阐述了 agent identity 模型如何将权限从"按用户分配"转变为"按频道分配"，从而让 AI 智能体在团队中安全、有效地工作。

核心问题是：当 AI 不再只是个人助手，而是团队协作的一员时，它应该以谁的身份去访问工具和文档？文章给出的答案是——AI 应该拥有自己的独立身份，而不是借用任何一个团队成员的权限。

核心概念讲解

从"单玩家"到"多玩家"AI

文章首先区分了两种 AI 使用模式：

  • 单玩家（Single Player）：一个人和一个 AI 助手对话，AI 借用这个人的账号权限去操作各种工具（如 Gmail、GitHub）。这是我们熟悉的个人助手模式。
  • 多玩家（Multiplayer）：多个人和同一个 AI 同时在一个共享空间协作，就像 Claude Tag 将 Claude 放在 Slack 频道中，与多位工程师同时工作。这时，"借用谁的权限"就成了问题——没有任何一个用户的身份能始终代表所有人。

Agent Identity 是什么

简单说，agent identity 就是给 AI 智能体创建独立的"账号"，让它以自己的身份去访问工具和系统：

  • 在 Slack 中发帖，显示为 Claude app 而不是某个用户
  • 在 GitHub 中操作，显示为 Claude GitHub App
  • 在数据仓库中查询，使用的是管理员创建的服务账号

这样设计的好处是：没有个人凭证混入共享频道，避免了通过共享频道窥探他人隐私的风险。

频道级权限与继承

Agent identity 模型的核心结构是：

  1. Workspace 级别：管理员定义一套基础的身份配置（连接哪些工具、哪些仓库、加载哪些技能）
  2. Channel 级别继承：每个频道默认继承 workspace 的基础配置
  3. 按需覆盖：管理员可以在特定频道调整权限，例如给工程频道开通 GitHub 和数据仓库权限，或将某个 CRM 连接限制在单一私有频道

文章主体讲解

开篇：引入问题

这一节讲什么

文章开头直接指出：要让 AI 智能体在人类-智能体团队中发挥最佳表现，它需要拥有与人类相同的工具、文档和上下文访问权限。

为什么重要

这是整篇文章的前提——如果 AI 无法获取必要的上下文，就无法真正成为团队的有效成员。但问题是：如何让 AI 获得这些访问权限？

关键点

  • 单人使用场景下，让 AI"扮演用户"是可行的
  • 多人协作场景下，这个模式行不通了

---

第一部分：为什么"扮演用户"失效了

这一节讲什么

作者从两个角度解释传统模式无法适用的原因。

为什么重要

理解问题根源，才能理解为什么需要全新的访问模型。

关键点

  1. AI 自主性不断增强：AI 能独立完成的任务长度大约每四个月翻一倍。现在的 AI 能自主安排未来任务、响应长期事件，甚至在提问者离线后继续工作。这意味着 AI 不能依赖"有人在场"来获得授权。
  1. 多玩家团队场景：Claude Tag 把 Claude 放到团队已有的工作空间中（比如三位工程师和产品经理一起 debug 的频道）。当多人同时与 AI 交互时，"该用谁的权限"没有唯一正确答案。

---

第二部分：Claude 以自己的身份行动

这一节讲什么

介绍 agent identity 模型的核心概念——Claude 拥有自己的独立账号。

为什么重要

这是解决方案的核心理念，解释了权限如何与个人用户解耦。

关键点

  • Claude 在 Slack 中以 Claude app 身份发帖
  • 在 GitHub 中以 Claude GitHub App 身份操作
  • 在数据仓库中通过管理员配置的服务账号查询

额外好处：没有个人凭证混入共享频道，共享频道永远不会成为窥探他人私人物品的"侧门"。

---

第三部分：权限继承机制

这一节讲什么

详细说明管理员如何在 workspace 和 channel 两个层级配置 AI 的能力。

为什么重要

这是实际配置时的核心操作逻辑，管理员需要理解继承和覆盖的工作方式。

关键点

管理员不仅配置凭证，还定义：

配置项说明
Repository accessClaude 可以读写哪些代码仓库
ConnectorsClaude 用来工作的工具和 API 密钥
Skills and pluginsClaude 动态加载的指令、脚本和资源，用于提升专业任务表现
Standing instructions每个频道的自定义指令和上下文

管理的便利性：撤销一个身份，该身份在所有地方的使用权限立即终止，比审计几十个用户账号下的智能体行为要简单得多。

---

第四部分：模型工作原理

这一节讲什么

将 agent identity 模型与传统 ACL（Access Control Lists）进行对比，并详细说明边界机制。

为什么重要

这是文章的技术核心，帮助读者理解模型的实际运作方式。

关键点

模型将问题从"这个用户能做什么？"转变为"这个智能体在这个隔离区能做什么？"

这意味着：没有直接访问仓库权限的频道成员，也可以让 Claude 去读取那个仓库——只要频道配置赋予了 Claude 该权限。

---

第五部分：身份边界如何工作

这一节讲什么

解释频道之间的隔离机制。

为什么重要

边界机制是安全性的保障，确保 Claude 在一个频道中学到的内容不会泄露到其他频道。

关键点

  • 每个私有频道有自己独立身份
  • 公开频道共享 workspace 级别身份
  • 法律频道的 Claude 身份无法触及未授权的代码
  • 工程频道的 Claude 身份无法读取未授权的法律文档
  • 记忆和访问都遵守这些边界：私有频道中学到的东西永远不会出现在更广泛的工作空间中

身份归属：身份属于频道，所以频道内任何人都可以默认 @Claude。管理员可以设置频道配置，仅覆盖权限最低的成员。在 Enterprise 计划中，RBAC 可以进一步限制谁能调用 Claude。

---

第六部分：广泛的默认访问

这一节讲什么

给出实际配置建议，说明应该给 Claude 什么样的初始权限。

为什么重要

这是管理员最需要知道的实践指导。

关键点

  • Claude Tag 的价值随工具和上下文访问的增加而复合增长
  • 连接的系统越多，Claude 越能跨系统组合信息——从 Slack 拉取线索、从 Drive 获取文档、从 tracker 提取工单、从仓库查询数据，组合成单一答案

建议策略：

  1. 从一开始给 Claude 慷慨的访问权限
  2. 根据组织偏好逐步收紧
  3. 从几个频道的基础配置开始
  4. 阅读审计日志
  5. 根据工作需求逐个谨慎地扩展权限

对于需要更细粒度控制的组织：可以在特定频道禁用 Claude Tag，也可以通过 RBAC 限制哪些用户可以使用。

---

第七部分：私信（Direct Messages）

这一节讲什么

说明 DMs 与共享频道在权限模型上的根本区别。

为什么重要

帮助用户理解什么场景该用 DM、什么场景该用频道。

关键点

  • DMs 运行在用户个人的 claude.ai 账号上
  • 使用的是用户的连接器、凭证，结果以用户的名义呈现
  • DMs 适合处理不应该出现在频道中的任务，比如邮件草稿或仅自己有授权的软件

---

第八部分：安全性与审计

这一节讲什么

介绍技术实现层面的安全保障。

为什么重要

企业采用需要明确的安保措施。

关键点

  • 凭证独立存储，映射到频道身份
  • 在网络边界请求时注入凭证
  • 未在白名单中的主机，流量直接被阻止
  • 所有例程、记忆写入、网络调用都记录在案
  • Claude 使用自己的服务账号行动，这些行为也会出现在各连接系统的独立日志中

---

结尾：展望未来

这一节讲什么

说明后续安全增强计划。

关键点

未来计划包括：

  1. Just-in-time 凭证授权：用户可以在特定敏感操作时临时授权，而不是永久扩大智能体权限范围
  2. 身份感知叠加层：在更复杂的权限结构组织中，在智能体范围之上叠加用户级别检查——只有频道配置和用户本人权限都允许时，Claude 才会行动

收束全文：从单玩家到多玩家 AI 的转变，使长期、团队协作的工作成为可能。Agent identity 确保 Claude 的访问权限足够广泛以发挥作用，同时又足够受限以在企业规模下保持安全。
